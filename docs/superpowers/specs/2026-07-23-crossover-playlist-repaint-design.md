# CrossOver 播放列表残影修复设计

## 背景

MusicPlayer2 的传统播放列表使用 `CListCtrlEx` 虚拟列表。搜索结果从多行缩减为一行“没有结果可以显示”时，CrossOver 会保留已经删除行的像素；改变窗口大小后残影立即消失，说明列表数据正确，问题出在客户区没有被完整标记为需要重绘。

`CListCtrlEx::OnEraseBkgnd` 从 2019 年起直接返回 `TRUE`，用于避免 Windows 下调整窗口大小时闪烁。该行为必须保留。

## 目标

- 列表数据替换后，CrossOver 能重画整个列表客户区，不留下已删除行的残影。
- 保留 Windows 下现有的背景擦除屏蔽和缩放防闪烁行为。
- 不引入 Wine 专用分支，不改变播放列表数据、搜索或绘制样式。

## 非目标

- 不重写播放列表控件。
- 不修改自绘播放列表。
- 不恢复全局 `WM_ERASEBKGND` 处理。
- 不加入同步绘制或强制刷新消息循环。

## 方案

修改 `CListCtrlEx::SetListData(ListData*)`：

1. 保存新的列表数据指针。
2. 通过 `SetItemCount` 更新虚拟列表项目数。
3. 调用 `Invalidate(FALSE)`，将整个客户区标记为需要重绘，但不请求背景擦除。

完整失效区域会让列表控件在下一次 `WM_PAINT` 中重画保留行和其余空白区域。`FALSE` 参数确保 `OnEraseBkgnd` 的防闪烁策略不变；异步失效允许连续搜索输入合并重绘，避免 `UpdateWindow` 带来的同步绘制和重入风险。

该逻辑对所有平台统一生效，因为“替换列表数据后重画可见内容”属于 `SetListData` 的正常契约，不是 CrossOver 专属业务行为。

## 测试与验证

### 自动化回归测试

在现有 Windows/MFC 测试能力允许的范围内，验证：

- 清除已有更新区域后调用 `SetListData`，列表整个客户区重新成为待绘制区域。
- 测试不要求背景擦除，确保修复依赖完整 `WM_PAINT`，而不是恢复 `WM_ERASEBKGND`。

如果当前 `UnitTest` 工程无法可靠创建 MFC 窗口控件，则使用一次性 Windows UI 回归程序执行同一断言，不为测试而增加生产代码接口。

### 构建验证

- 构建 Release x64。
- 构建 Release x86。
- 确认没有新增编译警告或错误。

### CrossOver 手工验证

1. 打开包含多首歌曲的传统播放列表。
2. 输入无法匹配任何歌曲的搜索词。
3. 确认只显示“没有结果可以显示”，下方没有旧行残影。
4. 改变窗口大小，确认画面没有发生额外内容变化。
5. 连续输入和删除搜索词，确认列表内容及时更新。

### Windows 回归验证

- 调整主窗口大小时列表不出现明显背景闪烁。
- 搜索、清空搜索、滚动和选择列表项保持原有行为。

## 风险与控制

- 全客户区重绘比局部重绘多绘制少量像素，但只发生在列表数据替换时，连续输入时系统还可以合并失效区域，成本可控。
- 不调用 `UpdateWindow`，避免同步刷新造成输入卡顿。
- 不修改 `OnEraseBkgnd`，避免重新引入已知缩放闪烁。
