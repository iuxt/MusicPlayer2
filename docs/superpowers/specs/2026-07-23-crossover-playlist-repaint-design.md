# CrossOver 播放列表残影修复设计

## 背景

MusicPlayer2 的传统播放列表使用 `CListCtrlEx` 虚拟列表。搜索结果从多行缩减为一行“没有结果可以显示”时，CrossOver 会保留已经删除行的像素；改变窗口大小后残影立即消失，说明列表数据正确，问题出在客户区没有被完整标记为需要重绘。

`CListCtrlEx::OnEraseBkgnd` 从 2019 年起直接返回 `TRUE`，用于避免 Windows 下调整窗口大小时闪烁。该行为必须保留。

第一次修复在 `CListCtrlEx::SetListData(ListData*)` 调用 `Invalidate(FALSE)`。实测表明该修复无效：它只把整个客户区加入更新区域，并明确不请求背景擦除。现有项目行会被重新绘制，但最后一行以下没有任何项目可以覆盖旧像素。

Wine 的 ListView 双缓冲实现还会在没有执行背景擦除时把当前窗口内容复制进后备缓冲区。应用随后只绘制仍然存在的项目，空白区域中的旧像素会继续进入最终画面。应用不能依赖不同 Wine、CrossOver 或 Windows ListView 版本隐式清理这块区域。

## 目标

- 列表数据替换后，CrossOver 能重画整个列表客户区，不留下已删除行的残影。
- 保留 Windows 下现有的背景擦除屏蔽和缩放防闪烁行为。
- 不引入 Wine 专用分支，不改变播放列表数据、搜索或绘制样式。
- 空白区域的清理发生在 ListView 的正常绘制周期内，不产生同步重绘或额外消息循环。

## 非目标

- 不重写播放列表控件。
- 不改变播放列表中现有项目的颜色、网格、选中或正在播放标记。
- 不恢复全局 `WM_ERASEBKGND` 处理。
- 不加入同步绘制或强制刷新消息循环。
- 不通过禁用 `LVS_EX_DOUBLEBUFFER` 绕过问题。

## 方案

### 空白区域绘制

在 `CListCtrlEx` 增加一个受保护的空白区域绘制函数。该函数：

1. 获取列表客户区。
2. 当列表存在项目时，获取最后一个项目的 `LVIR_BOUNDS`，并把待填充区域的顶部限制在最后一个项目底部和客户区边界之间。
3. 当列表没有项目时，使用整个客户区。
4. 使用 `m_background_color` 填充剩余的非空区域。

最后一个项目位于客户区底部之外时，计算结果为空，不执行额外绘制；最后一个项目位于客户区内时，只填充其下方，因此不会覆盖项目、选中背景或正在播放标记。

### 绘制时机

`CListCtrlEx::OnNMCustomdraw` 在控件级 `CDDS_PREPAINT` 返回：

```cpp
CDRF_NOTIFYITEMDRAW | CDRF_NOTIFYPOSTPAINT
```

保留现有逐项目自绘通知，并额外请求控件级 `CDDS_POSTPAINT`。收到 `CDDS_POSTPAINT` 后，使用通知提供的 HDC 调用空白区域绘制函数。

`CPlayListCtrl` 覆盖了 `NM_CUSTOMDRAW` 处理，因此也必须在它的 `CDDS_PREPAINT` 请求控件级后绘制，并在 `CDDS_POSTPAINT` 调用同一个基类函数。否则传统播放列表不会进入基类新增的分支。

控件级后绘制发生在项目和网格绘制之后、双缓冲内容提交到窗口之前。这样清理操作和项目绘制属于同一帧，不需要恢复 `WM_ERASEBKGND`，也不需要调用 `UpdateWindow`。

### 数据更新

保留 `SetListData(ListData*)` 中的 `Invalidate(FALSE)`。它仍负责保证数据替换会调度一次完整的异步绘制；新增的控件级后绘制负责在该次绘制中确定性地覆盖空白区域。

该逻辑对所有平台统一生效，因为“每次绘制后，非项目区域显示列表背景色”属于列表控件自身的绘制契约，不是 CrossOver 专属分支。

## 测试与验证

### 自动化回归测试

扩展现有源代码契约测试，验证：

- `SetListData(ListData*)` 在更新项目数后调用 `Invalidate(FALSE)`。
- `CListCtrlEx` 和 `CPlayListCtrl` 都在控件级预绘制阶段请求 `CDRF_NOTIFYPOSTPAINT`。
- 两个处理器都在控件级后绘制阶段调用同一个空白区域绘制函数。
- 空白区域绘制函数从最后一个项目底部开始，限制到客户区，并使用 `m_background_color` 填充。
- `OnEraseBkgnd` 仍直接返回 `TRUE`，没有恢复基类背景擦除。

当前本地环境不能创建 MFC ListView，因此源代码契约测试只负责防止关键绘制链路被意外删除；最终效果必须由 Windows 构建和 CrossOver 运行验证共同确认。

### 构建验证

- 构建 Release x64。
- 构建 Release x86。
- 确认没有新增编译警告或错误。

### CrossOver 手工验证

1. 打开包含多首歌曲的传统播放列表。
2. 输入无法匹配任何歌曲的搜索词。
3. 确认只显示“没有结果可以显示”，下方没有旧行残影。
4. 改变窗口大小，确认画面没有发生额外内容变化。
5. 连续输入和删除搜索词，确认列表内容及时更新。

### Windows 回归验证

- 调整主窗口大小时列表不出现明显背景闪烁。
- 搜索、清空搜索、滚动和选择列表项保持原有行为。
- 媒体库等其他 `CListCtrlEx` 列表的项目与空白背景保持正常。

## 风险与控制

- 每次列表绘制会额外填充最后一个项目以下的矩形。该区域原本就应显示统一背景色，填充成本与可见空白区面积线性相关，并受当前绘制频率限制。
- 基类和播放列表各有一个自绘处理器。两者调用同一绘制函数，避免两份边界计算产生差异。
- 填充发生在控件级 `CDDS_POSTPAINT`，不会覆盖已绘制项目；边界会限制在客户区内。
- 不调用 `UpdateWindow`，避免同步刷新造成输入卡顿。
- 不修改 `OnEraseBkgnd`，避免重新引入已知缩放闪烁。
- 不关闭双缓冲，保留现有滚动和缩放视觉效果。
